# 03-五文件映射建议

# AI 行动包, 社区参与 Community Participation

## 一、整体评估
这是一个非常适合 AI 先切入“运营协同层”的服务场景。

### 为什么适合
- 高频、重复、碎片化沟通很多
- 活动安排、worker brief、家属同步、记录整理都有固定骨架
- 每次服务信息量不大，但容易遗漏
- 需要持续复盘，天然适合结构化数据沉淀

### 为什么不能只靠 AI
- 现场支持仍高度依赖人
- 风险判断、身体协助、危机处理不能自动化替代
- 真实体验取决于 worker 稳定性和人际匹配

### 推荐定位
把 AI 定位成：
- 排期和沟通助手
- case note 整理助手
- 目标追踪助手
- 家属信息同步助手
- 服务边界提醒助手

而不是“自动做服务的人”。

---

## 二、Quick Start Guide

### 目标
用 7 天搭出一个最小可用版本，让团队先把“活动前确认 + worker brief + 活动后记录 + 家属摘要”跑顺。

### 最小上线范围
只做以下 4 个动作：
1. 生成活动前确认消息
2. 生成 worker 行前 brief
3. 把活动后口述整理成 case note
4. 生成家属/监护人摘要

### 推荐输入
- participant 基本资料
- 本次活动时间、地点、交通方式
- 本次目标
- 风险提醒与边界
- worker 活动后原始记录

### 推荐输出
- 给家属的确认消息
- 给 worker 的任务卡
- 内部记录 note
- 下次活动建议

### 成功标准
- 家属追问减少
- worker 临场摸索减少
- 记录完成率提升
- 月度回顾时能看见目标进展

## 三、Tool Guide

### Tool 1，Intake Structurer
用途：把首次咨询、家属聊天、历史备注整理成统一资料卡。

输入：
- 基本信息
- 兴趣偏好
- 不喜欢/触发因素
- 家属要求
- 风险与禁忌
- funding/服务限制

输出：
- participant one-pager
- 风险提示
- 沟通偏好
- 推荐活动方向

### Tool 2，Activity Planner
用途：生成本周活动建议和备选方案。

输入：
- 可服务时间
- 地点半径
- 预算
- 目标标签
- support worker 可用性

输出：
- 主方案
- 备选方案 A/B
- 交通与天气提醒
- 活动所需物品清单

### Tool 3，Worker Brief Generator
用途：活动前给 support worker 一页 brief。

输出建议包含：
- 今日目标
- 集合点与时间
- 沟通风格提醒
- 风险点
- 现场不要做什么
- 出现异常时联系谁

### Tool 4，Post-Visit Note Builder
用途：把 worker 的语音/碎片文字转成结构化 note。

建议字段：
- 实际活动
- 参与度
- 情绪状态
- 成功点
- 困难点
- 是否触达目标
- 是否需要跟进

### Tool 5，Family Update Generator
用途：生成发给家属的简短摘要。

原则：
- 友好
- 具体
- 不泄露不必要内部判断
- 风险内容分级表达

### Tool 6，Goal Tracker
用途：把多次活动记录聚合成周报/月报。

输出：
- 出勤趋势
- 活动类型分布
- 目标进展
- 高风险提醒
- 下阶段建议

## 四、Prompt Library

### Prompt 1，活动前家属确认
请根据以下资料，生成一条发给家属/监护人的活动前确认消息。要求简洁、友好、具体，包含时间、地点、交通方式、注意事项、如需变更如何联系。

### Prompt 2，worker 行前 brief
请根据以下 participant 信息和本次活动安排，生成一页 support worker brief。要包含今日目标、沟通偏好、风险提醒、边界提醒、集合信息、出现异常时的升级路径。

### Prompt 3，活动后记录整理
以下是 worker 的活动后原始记录，请整理成结构化 case note。要求区分事实、观察、风险、下次建议，不要夸大，不要写不必要判断。

### Prompt 4，家属活动后摘要
根据内部 case note，生成一段发给家属的活动后摘要。语气温和、具体、可读，重点写完成了什么、状态如何、下次建议，不要使用内部术语。

### Prompt 5，目标进展回顾
根据最近 4 周记录，输出目标进展总结。请判断哪些活动最有效，哪些时间段/场景更容易出问题，并给出下阶段建议。

### Prompt 6，服务边界提醒
根据以下服务类型与场景，生成一段内部提醒，说明本次支持范围、非支持范围、需要额外确认的事项，以及何时必须升级给主管或家属。

## 五、Workflow SOP

### SOP 1，新客进入
1. 收集基础资料
2. 整理 intake one-pager
3. 明确目标、预算、边界、紧急联系人
4. 生成首周建议活动
5. 指派合适 worker

### SOP 2，活动前 24 小时
1. 确认 participant 是否出席
2. 确认 worker 与交通
3. 发送家属确认消息
4. 发送 worker brief
5. 检查天气、门票、集合点、替代方案

### SOP 3，活动执行中
1. 到达签到
2. 若有迟到、取消、情绪变化，按级别同步
3. 若活动不适合继续，立即切换备选方案或结束支持

### SOP 4，活动后 15 分钟内
1. worker 提交语音/文字记录
2. AI 整理 case note
3. AI 生成家属摘要
4. 更新目标标签与下次建议

### SOP 5，每周回顾
1. 检查出勤和取消率
2. 看各 participant 目标进展
3. 看哪类活动最稳定
4. 看哪位 worker 需要补 brief 或培训

### SOP 6，异常升级
必须升级的情况包括：
- 参与者明显 distress 或安全风险
- 与服务边界冲突的新增要求
- 持续拒绝参与或环境明显不适配
- 家属投诉、交通事故、走失、伤害、重要财物问题

## 六、7-Day Plan

### Day 1
梳理现有服务流程，确定最小字段表。

### Day 2
搭建 intake 模板、worker brief 模板、case note 模板、家属摘要模板。

### Day 3
选 3 个真实或模拟案例试跑，从首次咨询到活动后总结跑通。

### Day 4
把服务边界、风险升级规则写成固定片段，嵌入模板。

### Day 5
让 1 到 2 位 worker 试用，收集“是否省事、是否清楚、是否太长”的反馈。

### Day 6
开始生成周回顾，检查目标记录是否真的可用。

### Day 7
确认是否进入常规运营，并决定下一步是接 CRM、表单、日历，还是先继续手工 + AI 半自动。

---

## 七、最终建议
优先级建议如下：
1. 先把记录标准化，再谈自动推荐
2. 先把家属沟通做稳定，再谈复杂 BI
3. 先覆盖一对一 community access，再扩展到小组活动
4. 先做低风险、可模板化场景，再碰复杂行为支持场景

如果只选一个切入口，我最推荐：
**活动后记录整理 + 家属摘要自动生成**。

因为它最容易落地，最容易马上省时间，也最容易让家属和管理者感受到价值。
